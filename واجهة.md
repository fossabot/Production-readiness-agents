الفجوات البنيوية التي ما زالت موجودة
1) لا يوجد Run Contract موحد

حالياً التدفق مرسوم، لكن غير واضح ما هو الشكل الرسمي لأي تشغيل.

أنت تحتاج نوعاً مركزياً مثل:

type CrewRunStatus =
  | "queued"
  | "starting"
  | "running"
  | "cancelling"
  | "completed"
  | "failed"
  | "cancelled";

type CrewRunRecord = {
  runId: string;
  repoPath: string;
  status: CrewRunStatus;
  startedAt: string;
  finishedAt?: string;
  durationMs?: number;
  selectedAgents: string[];
  modelConfigSnapshot: Record<string, unknown>;
  findingsSummary?: {
    critical: number;
    high: number;
    medium: number;
    low: number;
  };
  reportPaths?: {
    markdown?: string;
    json?: string;
    traces?: string;
  };
  error?: {
    code: string;
    message: string;
    details?: unknown;
  };
};

بدون هذا ستصبح HistoryPage و ProgressPage و ReportPage كل واحدة تبني “تصوراً خاصاً” للتشغيل. وهذه وصفة ممتازة للفوضى.

2) لا يوجد Event Schema للبث الحي

أنت كتبت:
TracingCollector يبث أحداث

هذا جيد، لكن ما هي هذه الأحداث بالضبط؟
من دون schema صريح ستتحول الرسائل إلى spaghetti IPC.

لازم تعريف موحّد مثل:

type CrewWorkerEvent =
  | {
      type: "run.lifecycle";
      runId: string;
      phase: "starting" | "running" | "completed" | "failed" | "cancelled";
      timestamp: string;
    }
  | {
      type: "agent.status";
      runId: string;
      agentId: string;
      status: "idle" | "running" | "completed" | "failed";
      startedAt?: string;
      finishedAt?: string;
      durationMs?: number;
    }
  | {
      type: "agent.finding";
      runId: string;
      agentId: string;
      finding: {
        id: string;
        title: string;
        severity: "Critical" | "High" | "Medium" | "Low";
      };
    }
  | {
      type: "trace.event";
      runId: string;
      traceId: string;
      spanName: string;
      timestamp: string;
      metadata?: Record<string, unknown>;
    }
  | {
      type: "run.error";
      runId: string;
      error: {
        code: string;
        message: string;
        details?: unknown;
      };
    };
3) لا يوجد Command Contract من الواجهة إلى worker

حالياً عندك فقط أسهم. هذا لا يكفي.

أنت تحتاج أوامر واضحة:

الأمر	الجهة	الغرض
crew:start-run	renderer → main → worker	بدء تشغيل جديد
crew:cancel-run	renderer → main → worker	إلغاء تشغيل
crew:get-run	renderer → main	جلب حالة تشغيل واحدة
crew:list-runs	renderer → main	جلب السجل
crew:get-report	renderer → main	قراءة التقرير
crew:update-settings	renderer → main	حفظ الإعدادات
crew:select-folder	renderer → main	فتح حوار اختيار مسار

ومقابلها قنوات بث:

القناة	الاتجاه
crew:run-event	main → renderer
crew:run-log	main → renderer
crew:run-progress	main → renderer
4) لا يوجد Cancellation Model

أي تطبيق إنتاجي لتشغيل وكلاء بدون إلغاء مضبوط هو لعبة شجاعة غير محمودة.

يجب أن يكون لديك:

AbortController لكل تشغيل

propagation للإلغاء من renderer إلى main إلى worker

تحديث الحالة إلى cancelling ثم cancelled

ضمان أن الإلغاء لا يفسد الـ store ولا يترك run نصف مكتوب

5) لا يوجد Error Taxonomy

أنت تحتاج تصنيفاً واضحاً للأخطاء، لا مجرد throw new Error().

النوع	أمثلة
CONFIG_ERROR	API key مفقود، نموذج غير صالح
INPUT_ERROR	repo path غير موجود
TOOL_ERROR	فشل أداة خارجية
AGENT_ERROR	وكيل فشل في مهمة
TRACE_ERROR	فشل التتبع
REPORT_ERROR	فشل توليد التقرير
STORE_ERROR	فشل الكتابة/القراءة من التخزين
WORKER_CRASH	thread انهار

7) SettingsPage يحتاج فصلًا أشد

الوصف الحالي يجمع:

نموذج كل وكيل

أدوات

مهارات

API keys

هذا كثير على صفحة واحدة ما لم تقسّمها تبويبياً:

التبويب	المحتوى
Agents	تمكين/تعطيل الوكلاء، النماذج
Models	provider + model + limits
Tools	الأدوات المسموحة لكل وكيل
Skills	skills محمّلة أو مسجلة
Secrets	API keys ومفاتيح البيئة
Runtime	trace, timeout, concurrency, retries
8) لا يوجد Policy Layer

هذا مهم جداً.
التطبيق لا يحتاج فقط إعدادات، بل سياسات تشغيل.

مثلاً:

type RuntimePolicy = {
  maxRunDurationMs: number;
  agentTimeoutMs: number;
  maxConcurrency: number;
  enableTracing: boolean;
  persistRawTraces: boolean;
  allowNetworkTools: boolean;
  allowFilesystemWriteOutsideRepo: boolean;
  autoOpenReportOnCompletion: boolean;
};

هذه ليست زينة. هذه ما يجعل التطبيق قابلاً للضبط والتحكم، لا مجرد صندوق غامض.

9) crew-worker.ts لا يجب أن يستورد فقط ../../dist/index.js بشكل أعمى

هذا سطر خطير بنيوياً إن لم تثبّت له واجهة adapter داخلية.

الأفضل:

// crew-runtime.ts
export interface CrewRuntime {
  run(input: CrewRunInput, emit: (event: CrewWorkerEvent) => void, signal: AbortSignal): Promise<CrewRunOutput>;
}

ثم يكون crew-worker.ts مجرد host، لا حشداً عشوائياً لمنطق التطبيق.

10) لا يوجد Report Artifact Contract

بما أنك اخترت سابقاً Markdown + JSON، فيجب أن يكون ذلك جزءاً صريحاً من التصميم، لا مجرد افتراض.

مثال:

type FinalReportArtifacts = {
  markdownPath: string;
  jsonPath: string;
  generatedAt: string;
  findingsCount: number;
  severitySummary: {
    critical: number;
    high: number;
    medium: number;
    low: number;
  };
};
